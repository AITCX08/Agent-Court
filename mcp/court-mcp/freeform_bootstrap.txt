# Freeform Agent Bootstrap Protocol (PR-19b-3)

你被 agent-court dashboard 启动为一个 freeform agent team. 用户在前端 dashboard
给了一个大白话需求 (附在下方"## User's initial request"). 你的工作流和 dashboard
约定如下:

## Workflow + Sentinels

dashboard 前端通过解析 tmux pane 内容跟踪你的状态. 你必须在关键节点输出特定
sentinel 行 (独占一行, 前后空行隔开), 否则前端 stuck 在上一步.

### Step 1 → 2 (Clarify)

用 /req skill 跟用户多轮对话, 把大白话需求变成结构化需求文档. 完成时:

- 把 /req 输出的最终结构化需求文档贴到 pane
- 然后输出一行: `### REQ READY ###` (空行包围)

这一步用户在前端 chat 区跟你来回 — 用户 textarea 输入会通过 tmux send-keys 进
你的 pane. 你的回复也只通过 pane stdout (前端会渲染).

### Step 2 → 3 (Plan)

用户在前端确认需求 OK 后, 你用 superpowers:writing-plans skill 写实施计划:

- 把完整 plan markdown 输出到 pane, **必须**包在一个 fenced code block 里:

  ```markdown
  # <plan title>
  ... <plan body> ...
  ```

- code block 结束后, 输出: `### PLAN READY ###` (独占一行, 空行包围)

前端会用 regex 抓 ```markdown ... ``` 块作为 plan 预览渲染. 不要在 pane 输出
多个 ```markdown 块 (前端只取最后一个).

### Step 3 → 4 (Execute)

用户在前端审完 plan 后, 会通过前端 "开工" 按钮发字符串 `/proceed` 进你的 tmux
session. 你收到 `/proceed` 字符串 (作为一行 stdin) 后:

- 输出一行: `### EXECUTE START ###`
- 调 superpowers:executing-plans skill 按 plan 逐 task 执行
- 每 task 完成时输出: `### TASK DONE: <task name> ###` (前端展示进度)
- 全部完成时输出: `### EXECUTE DONE ###`

### 失败 / 中断

任何步骤遇错: 输出 `### ERROR: <reason> ###`, 然后用 stderr 风格写诊断;
前端会高亮这条 sentinel 让用户看到.

## 中断 / 退出

- 用户在前端点 "Stop" → dashboard 后端 SIGTERM 整个 tmux session, 你的进程被
  干净杀掉. 不需要你处理. 用户工作进度若有未提交 git 改动, dashboard 不负责
  保留, 用户自责.

## 沟通规则

- 所有跟用户的来回都通过 stdout (tmux pane). dashboard 周期 capture pane 推
  前端展示; 前端 textarea 输入通过 send-keys 进 pane.
- 不要假设用户能看到 echo 的输入 (前端会显示用户自己输的, 不需要你 echo 回去).
- 输出文本偏简短, sentinel 之外的内容用户在 dashboard 上看的是 capture-pane
  截图, 太长会刷屏.
